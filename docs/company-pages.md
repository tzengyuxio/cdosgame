# 公司頁（content/companies）撰寫指南

`content/companies/<字串>.md` 是**手寫正本**（非生成物），檔名即遊戲資料中 `developer`/`publisher_tw` 用的公司字串，用來 join 聚合。本文定調公司頁的**框架、資料來源、行文風格與特殊例外**。

## 一、框架

### Frontmatter（schema 見 `src/content.config.ts`）

原則：**凡 infobox 可呈現、或頁面可結構化列出的資訊就放 frontmatter**；查證理由、取捨過程不放這裡（見「行文風格」）。

| 欄位 | 型別 | 說明 |
|---|---|---|
| `name_zh` | 必填 | 公司中文全名（如「智冠科技」）；未證實的全名用通稱（如「歡樂盒」非「歡樂盒資訊」） |
| `name_en` | 選填 | 官方英文名；**各源不一致或疑為 typo 時從缺**（勿塞推測值） |
| `aliases` | 陣列 | 簡稱、別名、英文俗稱 |
| `founded` / `dissolved` | 數字 | 成立／歇業（或被併走）年；衝突時取公司登記值 |
| `status` | enum | `active`/`defunct`/`acquired`/`merged`/`renamed` |
| `successor` | 選填 | 後繼實體（漢堂→智樂堂、宇峻→宇峻奧汀、光榮→光榮特庫摩） |
| `region` / `hq` | 選填 | 地區（台灣/日本…）／總部城市 |
| `founder` | 陣列 | 創辦人；不詳則從缺（90 年代小廠常見） |
| `roles` | enum 陣列 | `developer`/`publisher`/`localizer`/`distributor`——區分自製商 vs 代理／中文化商 |
| `website` | URL | 現行官網；已歇業多半從缺 |
| `featured_series` / `featured_games` | 陣列 | 代表系列字串 / 代表作 `cdg-NNNN`（須在該公司 developed/published 清單內才會顯示） |
| `sources` | URL 陣列 | provenance，頁面底部以「參考資料」列出 |

### Body（標準三段，保留彈性）

1. **簡介**（無標題，開頭段）：120–180 字，公司定位、創立、核心作品。
2. **## 沿革**：公司演變、併購、招牌系列、讀者會在意的釐清。可用敘事段落，**事項繁多時改用條列式更好讀**，不必拘泥文章形式。**無此類內容可省略**。
3. **作品列表**：由頁面自動聚合（開發作品／台灣發行代理），通常不必手寫；資料極豐者可另附產品目錄表（如大宇）。

資料很多可再加段落；資料少（如歡樂盒）僅簡介即可。**不強求一致**。

## 二、資料來源（依可靠度）

- **主**：中文維基百科、公司官網、MobyGames 公司頁。
- **次**：公司登記（twincn / 經濟部 / cnyes，查創立日民國年/地址/狀態）、Steam 開發者頁、巴哈姆特、PTT old-games、pixnet 老遊戲考據（「水色小築」是 PC-98 公司聖經）。
- **佐證**：Newton 百科、百度百科（易錯，僅交叉佐證）。
- **時事**：中央社／科技新報／iThome（近年併購、改名、上市櫃、IP 出售）。

⚠ 來源陷阱：維基部分條目「無參考來源」（漢堂）；Newton 會把年份帶錯（天堂鳥 1998/2004）；公司登記會把改名／同址的**不同實體**混在一起（歡樂盒 1996 ≠ 歡樂盒多媒體 2017）。

## 三、行文風格

語氣介於**維基百科與愛好者介紹站**之間：客觀敘事、好讀、有重點。

- **正文不寫研究筆記**：不要出現「已確認（多源一致）」「查無佐證」「採 1992 年」「frontmatter 不填」這類查證／編輯 meta。取捨結果直接反映在 frontmatter（如不確定就讓欄位從缺）＋ `sources`，理由留在 git commit。
- **讀者會在意的釐清用自然句子寫進沿革**，例如「玩家常把大宇的《天使帝國》誤記為漢堂作品」——這類屬考據趣味，是愛好者站的價值，要留。但**不要寫成「糾正」語氣**（公司介紹不需糾正誰），直接陳述事實即可。
- **不必提及「不詳」**：創辦人等資訊查無時，欄位與正文都直接略過，不要寫「創辦人不詳／查無」。

## 四、特殊例外（撰寫時對照）

1. **歷史歸屬 vs 現況**：記 DOS 時代的開發／發行。集團改名、併購、IP 出售（大宇 IP 2024 售出；漢堂→智樂堂；宇峻→宇峻奧汀；光榮→光榮特庫摩）只在沿革敘述＋ `status`/`successor`，**不改寫史實歸屬**。
2. **公司沿革變動：拆頁或繼承，看「性質 × catalog 是否有作品」**。頁面對應 catalog 實際使用的 developer 字串；早期作品掛在當年公司名下，就為該名建頁。三種情形：
   - **兩方合併** → 拆成「前身 A／前身 B／合併體」多頁，以 `predecessor`/`successor` 與正文 `/companies/<名>` 互連。例：**宇峻**（13 款）＋**奧汀**（10 款）→**宇峻奧汀**（11 款），三頁皆有作品故都建。
   - **改名／收購延續**（同一脈絡換名）→ **不拆，單頁，主名用原名**，後繼僅以 `successor` 文字標註。例：漢堂→智樂堂，留漢堂單頁。
   - **後繼／另一方在台早期幾無經營** → 只留主要那家。例：光榮＋Tecmo→光榮特庫摩，只建**光榮**（catalog developer 字串為 `KOEI`，61 款）；Tecmo 僅 2 款、特庫摩/光榮特庫摩無作品，皆不另開頁。
   
   通則：**successor/predecessor 指向的實體若在 catalog 無作品（或量極少且非台灣早期重點）就不開頁**，只留文字標註（前身/後繼在頁面會自動判斷：有頁才連結、無頁顯示純文字）。
2. **開發 vs 發行**：用 `roles` 區分。《金庸群俠傳》河洛開發／智冠發行 ≠ 智冠自製金庸改編；華義／歡樂盒／第三波以代理中文化為主（catalog 實證：華義 5 dev / 76 pub）。
3. **作品歸屬易混**：天使帝國＝大宇非漢堂；三國群英傳＝奧汀非宇峻；務必查清再寫。
4. **不確定即從缺**：英文名不一致（天堂鳥/歡樂盒/漢堂）、創辦人不詳（漢堂/天堂鳥/歡樂盒）、正式全名未證實（歡樂盒）→ 對應欄位留空，不臆測。
5. **年份衝突**：取公司登記值，必要時於沿革用一句自然帶過。
6. **解散主因要查實**：熊貓並非「A'Can 開發失利」而是 C&E（精訊）引擎侵權訴訟敗訴；A'Can 是 Funtech 旭展的主機。
7. **款級代理逐款查**：公司頁只作概述（如光榮台灣代理隨年代在第三波／台灣光榮／智冠間變動），個別遊戲代理商請在各 `cdg-NNNN` 條目標註。
8. **譯名並列**：同作不同譯名（摔角天使＝擂台美少女）在遊戲條目用 `title_aliases`。
